Volume Rendering
with
Oculus Rift
By Marco Godi
May 16, 2015
The goal of this project was
the creation of a software which
allows to experience
volumetric datasets in an
immersive way through the virtual
reality headset Oculus Rift
At the time of writing there is no
software that does something like
this, so we had to find our own
solution.
We used the Oculus SDK for the
Rift support, and VTK C++ library
for everything else.
A quick overview on Volume Rendering
Definition:
Volume rendering is a set of
techniques used to display a
2D projection of a 3D
discretely sampled data set,
typically a 3D scalar field.
Usually these datasets
are acquired by devices
such as CT, MRI and
X-Ray scanner and consist
of a sequence of 2D slice images.
A dataset is basically a 3D matrix of density values
(which represent every single voxel).
We need to define a transfer function to give RGBA values
to each density value. 
 
An indirect solution for volume rendering is through
isosurface extraction, that consists in grouping in different 
polygonal surfaces all the voxels in the same range of values.
While this is less computationally
intensive than direct rendering,
we discarded this solution
because it is not "true" volume
rendering (for example you
wouldn't be able to cross section
the dataset).
Volume Rendering Techniques
Ray Casting
  • Image based: a ray is generated from for each image pixel.
  • Derived from the rendering equation: the final RGB 
    result is the result of the accumulated RGBA values encountered
    by the ray after applying 
    the desired shading
  • Trilinear interpolation is
    used to determine the
    value of the sampling
    points (in general they
    don't intersect voxels).
  • Clipping by the volume
    boundaries to save time.
  • Very high quality
  • high resource cost
Splatting
  • Faster at the cost of lower quality.
  • Each voxel is "splatted" onto the viewing
    surface in a "back to front" order.
  • The splats are rendered as disks.
Shear warp
  • Faster, less accurate and with a meaningful memory overhead.
  • Multiple copies of the volume are stored and used for different
    orientations and
    scalings (at a faster
    speed because of the
    now more favorable
    pixel/voxel alignment)
Texture-based volume rendering
  • In earlier years, dedicate 3D texture mapping systems were used
  • Nowadays, PC video cards can do it efficiently and allow
    to program the GPU to suit our specific needs
  • Reasonable quality, noticeable "glitches" during rotation
The common solution for direct volume rendering (which is the one
our software uses) is a hardware-accelerated raycasting, which
consists in taking advantage of the general purpose nature of the
modern GPUs.
Basically a pixel shader is used to implement the volumetric
raycasting algorithm. Because of the MIMD nature of recent GPUs
algorithm steps can be executed in a parallel way, speeding up a lot
the process.
Which one should you choose?
Some Optimizations...
  • Skip empty spaces (completely transparent)
  • Terminate rays earlier when you encounter density above a
    certain threshold
  • Octree space subdivision (better compression and lookup speed)
  • Less interesting regions can be represented in lower detail...
  • ... or the whole dataset too. Even in our software it's possible to
    set a reduction factor that determines the ratio of voxels to be
    shown
What we did
As mentioned before, we relied upon VTK libraries for our volume
rendering, in particular we used vtkSmartVolumeMapper class
with DefaultRenderMode: It will use GPU raycasting if supported.
If not it will Texture based rendering for interactions and software
raycasting for still renders. If not even 3D texture mapping is
supported, it will use only software raycasting.
A quick overview on Oculus Rift

"The Oculus Rift requires the scene to be rendered in split-screen stereo
with half the screen used for each eye. When using the Rift, the left eye
sees the left half of the screen, and the right eye sees the right half.
Although varying from person-to-person, human eye pupils are
approximately 65 mm apart. This is known as interpupillary distance
(IPD). The in-application cameras should be configured with the same
separation..."
"...This means that your application will need to render the  entire scene
twice, once with the left virtual camera, and once with the right."
From the official "Oculus Developer Guide"
The lenses of the Oculus provide a very large FOV (more than
90° horizontally) for enhanced immersion.
However, this causes a significant
distortion; in particular the distortion is a
pincushion distortion.
To counteract this distortion you need
a postprocessing with an equal and 
opposite barrel distortion so the two of
                                            them will cancel each other out.
                                            You also need to correct a color separation
                                            effect on the edges of the lens called
                                            chromatic aberration. 
As mentioned before, you need two
cameras because you have to render
two slightly different scenes every
frame.
These two cameras are shifted
sideways, but the projection axes
should be parallel to each other and
point in the same direction.
The optimal distance between the
two cameras depends on the person
which is wearing the HMD.
In fact, the Oculus Configuration
Utility allows to define user profiles
with personalized settings and tests
to determine the optimal IPD.
Head Tracking is achieved through the use of sensors, including
gyroscopes, accelerometers and magnetometers. The collective
information from all the sensors on the Oculus Rift is combined 
through a process known as sensor fusion.
With DK2 (at the time of writing  this is the most recent version)
Oculus Rift is able to track head orientation and head position.

Head position is maintained as a unit
quaternion or eventually in
yaw-pitch-roll form.
Head tracking
Head position tracking is achieved through 
a standard USB webcam with a infrared
filter that picks up the blinking frequencies
of the LEDs mounted all over the headset.
Each of these LEDs blinks at different
frequences so the software can distinguish
them from each other and calculate head
position.
The OculusSDK provides 
parameters such as the
distance of the frustum faces
from the camera to allow
the programmer to have a
visual representation of the
tracking frustum if needed.

On the image are shown the
default origin and yaw
orientation of the camera,
but these parameters can be changed at run-time to match the
current ones of the user.

Position tracking may stop working when: the headset exits the
frustum volume, the headset faces away from the camera, the
headset is occluded in some way or the headset moves too fast.
